# Calendar + Docs visual language mini-spec

Дата: 2026-04-21  
Статус: design spec / critique  
Scope: только visual language, без code changes

## 1. Visual language для calendar states

### Цель
Календарь должен читаться за 2 секунды: сначала тип дня, потом наличие факта. Не наоборот.

### Базовое правило
У дня только 2 визуальных слоя:
1. **Состояние дня**: Д / Н / В / праздник.
2. **Факт**: отдельный маркер поверх, но не замена основного состояния.

### Словарь состояний
- **Д**: самый спокойный позитивный сигнал, светлый зелёный акцент.
- **Н**: более заметный холодный акцент, но без «аларма».
- **В**: нейтральный серо-синий, визуально самый тихий.
- **Праздник**: не отдельный агрессивный цвет на весь день, а тонкий дополнительный warm-marker или outline, чтобы не спорить с Д/Н/В.
- **Факт**: не перекрашивает весь день. Только маленький устойчивый marker/dot/chip, одинаковый во всех сценариях.

### Правила композиции
- В ячейке один главный знак, это буква состояния.
- Число дня всегда слабее буквы состояния.
- Факт всегда рисуется одинаково, независимо от того, это Д, Н или праздник.
- Если есть и план, и факт, пользователь читает: **сначала состояние дня, потом факт наличия записи**.
- Нельзя кодировать разные смыслы одним и тем же цветом в соседних сценариях.

### Критика текущего направления
- Легенда в целом уже близка к минимализму, но **факт лучше держать как отдельный маркер, а не как конкурирующий полноценный цвет состояния**.
- Ночь и факт должны быть визуально различимы даже в быстром периферийном скане.
- Праздничность не должна выглядеть «важнее работы», иначе календарь начнет шуметь.

## 2. Visual language для docs entry

### Цель
Первый экран docs должен отвечать на вопрос: **что открыть сейчас**, а не показывать красивый каталог.

### Пропорции карточек
- Карточка вертикально компактная, ощущение пропорции примерно **1 : 2.6-3.2** по высоте к ширине строки.
- Внутри 3 зоны: маленькая иконка слева, основной текстовый стек по центру, тихая стрелка справа.
- Главный объём воздуха должен быть у текста, не у декора.

### Роль иконки
- Иконка только для быстрого различения трёх категорий.
- Иконка не должна быть главным фокусом карточки.
- Иконка живёт как маленький опознавательный маркер в отдельной мягкой плашке.
- Если карточка понятна без иконки, значит иконка выбрана правильно.

### Subtitle / note
- Subtitle максимум **1 строка**, в крайнем случае 2 очень короткие.
- Оптимальная длина: **35-55 символов**.
- Subtitle должен объяснять различие категории, а не пересказывать её название другими словами.

### Когда нужен серый текст
Серый текст нужен только для:
- eyebrow,
- subtitle,
- вторичной поясняющей строки,
- неактивной стрелки или вторичного nav.

### Когда серый текст не нужен
Не делать серым:
- название категории,
- ключевое действие выбора,
- то, что пользователь должен считать первым.

### Критика текущего направления
- Общая структура карточек удачная: icon + content + arrow работает.
- Но subtitle нельзя раздувать до почти описательного абзаца, иначе карта теряет мгновенное различие.
- Eyebrow полезен только пока он короче и тише title. Если он начинает конкурировать, его нужно упрощать.
- Стрелка должна оставаться служебной, не декоративной доминантой.

## 3. Итог
И в календаре, и в docs entry продукт выигрывает от одного правила: **один главный сигнал на один объект**.  
Для календаря это тип дня. Для docs card это название категории.
