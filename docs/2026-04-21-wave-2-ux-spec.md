# Wave 2 UX spec, home + quick-add + nav/docs

Дата: 2026-04-21  
Статус: draft for implementation  
Scope: только UX-спецификация волны 2, без смены data model

## 1. Цель волны

Упростить первый пользовательский сценарий после открытия приложения:
1. быстро понять, что сегодня происходит,
2. за 1-2 тапа перейти к записи смены,
3. не путаться между экранами, действиями и входом в документы.

Принцип: home отвечает на вопрос "что у меня сейчас", add отвечает на вопрос "как быстро записать смену", docs отвечает на вопрос "что мне открыть по задаче".

---

## 2. Новая иерархия главной

## 2.1. Основной порядок блоков

Порядок сверху вниз:

1. **Hero-блок "Сегодня / ближайшая смена"**
2. **Primary actions row**
3. **Календарь работы**
4. **Короткая сводка за месяц**
5. **Вторичные входы**: журнал смен, документы/инструкции

Что уходит ниже первого экрана:
- месячные агрегаты,
- вторичные информационные карточки,
- любые повторяющие блоки со сменами,
- вход в docs как отдельный равноправный приоритет с основной работой.

## 2.2. Hero-блок "Сегодня / ближайшая смена"

### Содержимое
- Заголовок состояния:
  - `Сегодня смена`
  - `Сегодня выходной`
  - `Ближайшая смена завтра`
  - `Смена идет сейчас`
- Главная карточка факта или плана:
  - дата,
  - интервал времени,
  - тип (факт / по графику),
  - короткие ключевые детали,
  - если есть факт, использовать максимально нативный reuse shift card.
- Один главный CTA:
  - если смены на сегодня нет: `Записать смену`
  - если есть план без факта: `Записать по плану`
  - если есть факт: `Открыть смену`

### Поведение
- Hero всегда один, без конкурирующих карточек рядом.
- Если на день несколько фактических записей, hero показывает агрегированное состояние и CTA `Открыть журнал за день`.
- Если есть и план, и факт, факт визуально главный, план показан вторичной строкой `По графику было: ...`.

## 2.3. Primary actions row

Сразу под hero, 2 компактные кнопки:
- `+ Смена`
- `Журнал`

Опционально третья, только если помещается без перегруза:
- `Документы`

Правило: это именно быстрые входы, не аналитика и не summary.

## 2.4. Календарь работы

Календарь остается главным вторым блоком, но:
- визуально ниже hero по приоритету,
- не спорит за внимание с CTA записи,
- легенда компактная,
- под календарем максимум одна строка пояснения про график/факт.

## 2.5. Сводка за месяц

Месячная сводка остается полезной, но вторичной:
- компактная карточка,
- без доминирующей типографики,
- без ощущения, что это главный сценарий home.

### Acceptance criteria, home hierarchy
- На первом экране до scroll пользователь видит hero и явный CTA записи смены.
- На home нет двух разных крупных блоков, конкурирующих за роль "главной смены".
- Календарь находится выше месячной аналитики, но ниже hero и primary actions.
- Вторичные входы не ломают reading order первого экрана.
- Состояния `нет смены`, `есть план`, `есть факт`, `несколько фактов` имеют разные тексты и CTA.

## 3. Quick-add first для смены

## 3.1. Принцип формы

Экран добавления делится на 2 слоя:

### Слой A. Обязательный быстрый ввод
Показывается сразу:
- дата,
- время начала,
- время окончания,
- тип записи,
- кнопка `Сохранить`

### Слой B. Детали
Скрыт под явным раскрытием `Добавить детали`:
- маршрут,
- локомотив,
- вагоны/оси/вес,
- топливо,
- заметка,
- прочие технические поля.

## 3.2. Поведение quick-add

### Базовый сценарий
1. Пользователь открывает add.
2. Сразу видит только core fields.
3. Может сохранить запись без обязательного прохода через длинную форму.
4. После сохранения получает мягкий post-save entry:
   - `Добавить детали`
   - `Готово`

### Сценарий из календаря
- Если вход из календаря/плана, дата и рекомендованный шаблон времени уже заполнены.
- Заголовок объясняет контекст: `Запись смены на 21 апреля`.
- Если день по графику выходной, copy не конфликтует с действием: `Можно записать фактическую смену, если работали вне графика`.

## 3.3. UI-блоки и состояния

### В верхней части формы
- краткий заголовок: `Новая смена`
- вторичная строка контекста:
  - `Быстрая запись, детали можно добавить позже`

### Core fields card
Содержит только:
- дата,
- начало,
- окончание,
- тип записи,
- primary CTA `Сохранить смену`

### Details disclosure block
- collapsed state: одна строка `Добавить детали`
- expanded state: полноценные дополнительные секции
- после раскрытия пользователь может сохранить внизу тем же основным CTA, без второго конкурирующего сценария

## 3.4. Copy rules
- Не использовать технические формулировки уровня "основные/расширенные параметры".
- Использовать человеческие формулировки:
  - `Добавить детали`
  - `Можно заполнить позже`
  - `Записать смену`
- Если обязательных полей не хватает, ошибка показывается рядом с конкретным полем, а не общим тревожным блоком сверху.

### Acceptance criteria, quick-add first
- Пользователь может сохранить смену, не проходя через длинный список технических полей.
- На первом экране формы не более 4-5 смысловых элементов до primary CTA.
- Дополнительные поля не исчезают из продукта, а переносятся в раскрываемый слой.
- Вход из календаря предзаполняет дату и контекстный шаблон времени.
- После быстрого сохранения есть понятный путь "добавить детали позже".

## 4. Логика nav / docs entry

## 4.1. Правило навигации

Bottom nav должен содержать только **экраны**, а не смешение экранов и одноразовых действий.

Рекомендуемая логика:
- `Главная`
- `Журнал`
- `Добавить`
- `Документы`

Если `Добавить` остается центральной FAB/action, он не должен визуально конкурировать с экранными табами и должен читаться именно как действие.

## 4.2. Рекомендуемый вариант

Предпочтительный вариант для волны 2:
- в bottom nav оставить 3 экранных раздела: `Главная`, `Журнал`, `Документы`
- `Добавить смену` сделать отдельной центральной FAB над nav

Почему:
- пользователь ясно отличает "куда перейти" от "что сделать сейчас",
- quick-add становится первичным действием без засорения структуры экранов,
- docs не выглядит случайной тех. вкладкой наравне с action.

## 4.3. Docs entry logic

Вход в docs должен описывать пользовательскую задачу, а не внутреннюю категоризацию файлов.

### Entry screen docs
Верхний уровень docs показывает 3 task-based entry cards:
- `Инструкции`
- `Памятки`
- `Папки / документы`

Под заголовком краткая helper-copy:
- `Откройте нужный материал по типу задачи`

### Внутри категории
- список документов,
- поиск,
- recent/opened last при наличии,
- офлайн-статус как вторичная метка, а не основной акцент.

## 4.4. Routing rules
- Из home entry `Документы` ведет на верхний task-based docs hub, не сразу в последнюю тех. вкладку.
- Если пользователь зашел в docs из конкретного контекста позже, допустим direct deep-link в нужную категорию.
- Приложение должно помнить последнюю открытую категорию только внутри docs-сессии, но основной entry всегда остается понятным и стабильным.

## 4.5. States for docs entry

### Empty/default state
- Показываются 3 крупные карточки категорий.
- Ни один технический tab id не виден пользователю.

### Search state
- Поиск работает поперек текущей выбранной категории.
- Если результатов нет, сообщение формата `Ничего не нашли, попробуйте другую категорию или запрос`.

### Returning state
- Можно показать маленький блок `Недавно открывали`, но ниже основных task-entry карточек.

### Acceptance criteria, nav/docs
- Bottom nav визуально не смешивает screen navigation и primary action без явного различия.
- `Добавить смену` воспринимается как быстрый action first.
- Docs entry начинается с task-based выбора, а не с тех. внутренних категорий/табов.
- Пользователь может попасть в документы с home и понять следующий шаг без обучения.
- Повторный вход в docs не ломает понятный default entry.

---

## 5. Границы реализации

В эту волну не входят:
- полная переработка data model,
- глубокая переработка docs backend/storage,
- новый отдельный onboarding flow,
- радикальная перестройка календарной логики.

В эту волну входят:
- новая визуальная иерархия home,
- quick-add first layout/copy/states,
- разделение screen nav и action logic,
- task-based first entry для docs.

---

## 6. Definition of done для wave 2 UX

Волна считается принятой, если:
1. home за 3 секунды отвечает, что происходит сегодня и куда нажать для записи;
2. смену можно записать через короткий сценарий без перегрузки деталями;
3. навигация не путает экраны и действия;
4. вход в docs понятен по задачам пользователя, а не по внутренней структуре контента;
5. все описанные состояния покрыты UI без немых или двусмысленных переходов.
